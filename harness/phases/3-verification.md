# 阶段 3：验收

三个顺序子阶段：代码审查 → QA 测试 → 测试代码审查。每个依赖前一个的输出。

## 子阶段 3.1：代码审查

Spawn Code Reviewer，提供：角色定义 + spec.md + 技术方案 + 所有生产代码 + config.md。

审查维度：正确性、安全（注入、认证绕过、输入校验）、性能（N+1 查询、不必要分配）、错误处理、风格。

产出：`reviews/code.md`，问题分严重/重要/轻微。

如有严重问题 → 分配给责任 Agent 修正。修完后重新跑 CI。

## 子阶段 3.2：QA 测试

Spawn QA Runner，提供：角色定义 + test-matrix.md + spec.md + 技术方案 + 所有代码 + config.md。

任务：写测试代码（匹配 test_file_suffix）、跑测试、出报告。

产出：测试代码文件 + `reviews/qa.md`（含测试执行结果、失败分析、覆盖率报告）。

如有测试失败 → 确定归属（测试 bug 还是代码 bug）→ 分配给责任 Agent 修正。最多 2 次。

## 子阶段 3.3：测试代码审查

Spawn Code Reviewer（聚焦测试代码），提供：角色定义 + test-matrix.md + qa.md + 所有测试代码。

审查维度：覆盖率完整性、测试正确性、边界情况覆盖、假阳性（永远通过的测试）、断言质量。

产出：`reviews/test-code-review.md`。

## 关卡 #3：最终验证

通过条件：
- 所有审查报告中 0 个严重问题
- 全部测试通过（0 失败）
- 覆盖率 ≥ 配置阈值
- qa-log.md 中无遗留问题

如果失败：汇总 → 分配 → 修 → 重跑 CI → 重跑受影响阶段 → 重验。最多 2 轮。
