# 阶段 1.5：技术方案（并行）

API Implementer 和 Biz Implementer **同时**工作。各自产出独立的方案文档，Lead 之后交叉检查一致性。

## 并行 Spawn

### API Implementer 的输入
角色定义 + config.md（尤其是 api_layer_pattern）+ spec.md + 代码库概览

### Biz Implementer 的输入
角色定义 + config.md（尤其是 biz/dao_layer_pattern）+ spec.md + 代码库概览

## API Implementer 产出：`tech-design/api.md`
- 文件结构、Handler 签名（带类型）、路由注册、请求校验方案、错误响应格式
- 记录调用哪些 Biz 接口

## Biz Implementer 产出：`tech-design/biz.md`
- Service 接口签名、DAO 接口签名、业务逻辑流程、数据模型定义、文件结构

## Lead：一致性检查（关卡 #1.5）

两个 Agent 都完成后：

1. **检查接口对齐**：API 方案中调用的每个 Biz 接口在 Biz 方案中都存在？参数类型匹配？返回类型匹配？
2. **检查文件冲突**：有没有两个 Agent 声称要写同一个文件？
3. **写入契约文档**：`docs/harness/{功能名}/contracts/api.md`——汇总接口定义 + 文件归属表
4. **解决冲突**：接口不匹配 → 两边对齐（通过 qa-log.md）；文件冲突 → Lead 裁决归属

最多 2 次修正尝试。
