# 阶段 1：需求分析

Planner 分析 PRD，产出结构化需求 + API 契约。

## 输入给 Planner

Lead 提供：本阶段文件、Planner 角色定义、config.md、PRD 文本、第 0 步的代码库概览。

## Planner 的任务

### 1. 分析 PRD
- 识别所有功能需求
- 排优先级：P0（必须有）、P1（应该有）、P2（锦上添花）
- 识别非功能性需求
- 标注任何含糊或缺失的信息

### 2. 定义 API 契约
每个接口：HTTP 方法 + 路径、请求参数、请求体结构（带类型）、响应体结构（带类型）、错误响应、认证要求。

### 3. 识别影响的文件
- 哪些文件要 CREATE（在 api/biz/dao 各层）
- 哪些文件要 MODIFY
- 哪些文件受影响但不改

### 4. 数据模型变更
- 新表/集合、现有数据变更、迁移说明

## 产出：`docs/harness/{功能名}/requirements/spec.md`

结构：概述 → P0/P1/P2 功能需求表 → API 契约（每接口含请求/响应/错误）→ 数据模型 → 文件清单 → 非功能性需求。

## 问答协议

Planner 遇到含糊时：不要猜 → 写问题到 qa-log.md → 告知 Lead。格式：
```
### [Planner] {问题概述}
**上下文**：{PRD 哪里不清楚}
**问题**：{具体哪里不确定}
**影响**：{答案如何影响规格}
```

## 完成标准

- [ ] PRD 中每个功能领域至少对应一项 FR
- [ ] 所有 P0 FR 有可衡量的验收标准
- [ ] 所有 API 接口有完整的请求/响应结构
- [ ] 文件清单完整
- [ ] qa-log.md 中无 Planner 未解决的问题
