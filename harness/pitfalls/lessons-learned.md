# 踩坑经验

> 运行 Agent Team Harness 过程中积累的经验。
> 欢迎添加你自己的发现。

---

## 上下文管理

### 坑：大 PRD 导致 Agent 上下文溢出
**症状**：Agent 产出不完整、漏需求、输出截断。
**原因**：PRD + spec + 代码库概览超出了 Agent 的有效上下文窗口。
**修复**：
- 大 PRD（>2000 字）让 Lead 先摘要再传给 Agent
- 大功能拆成子功能，跑多次 `/harness-dev`
- Planner 应该产出精炼的 spec，不是 PRD 的拷贝

### 坑：长并行阶段后丢失上下文
**症状**：并行 spawn API + Biz + QA Designer 之后，Lead "忘记"了之前的决策。
**原因**：Lead 的上下文被 Agent 产出和进度更新填满。
**修复**：
- Lead 在每个阶段后在 PROGRESS.md 中写简短摘要
- 关卡检查前，Lead 重新读相关文档
- 阶段文件短小精悍 — Lead 按需加载

## 关卡执行

### 坑：关卡形同虚设
**症状**：关卡 #1 "通过"了但后续阶段发现需求缺失。
**原因**：Lead 没有严格逐项核查检查清单。
**修复**：
- 把 `harness/gates.md` 当作逐字检查清单——逐项打勾
- 不确定就升级给人类
- 宁可关卡不通过返工，也比交付残缺功能强

### 坑：CI 失败归属错误
**症状**：编译失败，Lead 让错误的 Agent 去修，返工周期白费。
**原因**：Lead 没有仔细解析错误输出。
**修复**：
- CI Runner 的报告应该包含"可能责任人"字段
- 用错误文件路径匹配 config 中的 pattern
- 不确定就两个 Agent 都查——不要猜

## Agent 通信

### 坑：Agent 猜测而不提问
**症状**：Agent 产出与 PRD 意图不符。
**修复**：Agent 角色文件中明确写"如果不确定，写 qa-log.md 问——不要猜"。Lead 在进入下一阶段前检查所有 qa-log.md 问题已解决。

### 坑：QA log 刷屏
**症状**：qa-log.md 被琐碎问题填满。
**修复**：Agent 角色文件定义"决策阈值"——影响行为的才问，命名/风格自己定。Lead 可以批量回答琐碎问题。

---

## 会话恢复

### 坑：PROGRESS.md 过期
**症状**：会话恢复后从错误的步骤继续。
**修复**：每步完成后立即更新 PROGRESS.md——不要攒着一起更新。如果怀疑 PROGRESS.md 过期，检查文件时间戳。

### 坑：恢复后不读历史上下文
**症状**：恢复的会话做出与之前矛盾的决策。
**修复**：恢复流程：1) 读 PROGRESS.md → 2) 读所有之前的阶段产出 → 3) 继续。Lead 在 PROGRESS.md 中写"交接上下文"节。

## 文件归属

### 坑：两个 Agent 改了同一个文件
**症状**：API Implementer 和 Biz Implementer 都写同一个 model 文件。
**修复**：关卡 #1.5 明确检查文件冲突。contracts/api.md 的文件归属表必须列出每个文件。规则：两层共用的 model/type → 归 Biz（API 层 import）。

---

## 复杂度评估

### 坑：低估复杂度
**症状**：评估为 S/M 结果发现是 L，团队太小。
**修复**：第 0 步必须在代码库里 grep 相关 pattern，不能只读顶层目录。M 和 L 之间犹豫就选 L。阶段 1 后可以重新评估。

## 性能

### Token 消耗估算
| 复杂度 | 典型 Token | 典型耗时 |
|--------|-----------|---------|
| S（简单） | ~50K | ~10 分钟 |
| M（中等） | ~150K | ~25 分钟 |
| L（大型） | ~300K+ | ~45+ 分钟 |

### 优化建议
- S 功能跳过 Planner，Lead 直接写需求
- Token 预算紧时 M 功能可合并 CI Runner + Code Reviewer
- 诚实评估复杂度——不要用 L 流水线跑 S 功能
