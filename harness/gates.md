# 质量关卡

每道关卡是强制的检查点。Lead spawn **Gatekeeper**（独立 Agent，`agents/gatekeeper.md`）执行对抗性检查，产出 Gate Report。Lead 审阅报告并决定：通过（继续）或失败（返工）。每道关卡最多 2 次返工。2 次后仍失败则升级给人类用户。

Gatekeeper 的核心价值：Lead 既是执行者又是检查者时存在利益冲突（Lead 倾向于"放过"以推进进度）。Gatekeeper 没有执行责任，唯一使命是找问题，提供了 Lead 做不到的对抗性视角。

---

## 关卡 #1：需求质量

**位置**：阶段 1（需求分析）之后 | **评估者**：Gatekeeper（执行检查）+ Lead（裁决）

### 检查清单

- [ ] **覆盖**：PRD 中每个功能领域至少对应一项功能需求（FR）
- [ ] **可衡量**：所有 P0 需求有明确、可衡量的验收标准（不能是"能正常工作"这种模糊话）
- [ ] **API 完整**：每个接口包含 HTTP 方法、路径、请求结构（带类型）、响应结构（带类型）、错误码
- [ ] **文件清单**："文件清单"节列出了所有要创建/修改的文件，归属清晰
- [ ] **数据模型**：如需数据变更，"数据模型"节已填写
- [ ] **无未解决问题**：qa-log.md 中没有 Planner 未回答的问题

### 常见失败及修复
| 症状 | 修复 |
|------|------|
| 缺少 API 接口 | 问 Planner："PRD 提到了 {功能} 但没有对应接口。请补充。" |
| 验收标准模糊 | 问 Planner："FR-{N} 写'能正常工作'。请给出可衡量的标准。" |
| 请求/响应缺类型 | 问 Planner："请给 {接口} 的结构体加上明确类型。" |
| 文件清单不完整 | 问 Planner："{组件} 的代码放哪里？更新文件清单。" |

## 关卡 #1.5：技术方案一致性

**位置**：阶段 1.5（技术方案）之后 | **评估者**：Gatekeeper（执行检查）+ Lead（裁决）

### 检查清单

- [ ] **接口对齐**：API 层调用的每个 Biz 接口在 Biz 方案中都有定义
- [ ] **类型一致**：API → Biz 调用的参数类型与 Biz 接口定义匹配
- [ ] **返回类型一致**：Biz → API 的返回类型与 API handler 期望匹配
- [ ] **无文件冲突**：没有两个 Agent 声称要写同一个文件
- [ ] **路由覆盖**：spec.md 中的每个接口在 API 方案中有对应路由
- [ ] **Service 覆盖**：每个业务操作在 Biz 方案中有对应 service 方法

### 常见失败及修复
| 症状 | 修复 |
|------|------|
| API 调 `UserService.Get()` 但 Biz 定义的是 `Fetch()` | 两边对齐方法名——Lead 裁决 |
| 两个 Agent 都声称改 `model/user.go` | Lead 判给 Biz 层（model 归 biz） |
| API handler 期望 `(User, error)` 但 Biz 返回 `(*User, bool, error)` | 对齐返回类型——spec.md 是权威依据 |

## 关卡 #2：编译与 Lint

**位置**：阶段 2（编码）之后 | **评估者**：CI Runner（执行）+ Lead（裁决）

### 检查清单

- [ ] **编译**：退出码 0，无错误
- [ ] **Lint**：0 错误（项目已有的存量警告可以放行）
- [ ] **Proto**：（如启用）退出码 0，无错误
- [ ] **无新增警告**：本次功能没有引入新的 lint 警告

### 失败归属

解析错误输出确定责任：
- 文件在 `api_layer_pattern` 下 → API Implementer
- 文件在 `biz_layer_pattern` / `dao_layer_pattern` 下 → Biz Implementer
- 涉及两层 → 看哪个 Agent 的代码报错
- 不明确 → 两个 Agent 都检查，通过 qa-log.md 协调

## 关卡 #3：最终质量

**位置**：阶段 3（验收）之后 | **评估者**：Gatekeeper（执行检查）+ Lead（裁决）

### 检查清单

- [ ] **0 个严重问题**：代码审查（3.1）中无 CRITICAL
- [ ] **0 个严重问题**：测试代码审查（3.3）中无 CRITICAL
- [ ] **全部测试通过**：QA 报告（3.2）中 0 失败
- [ ] **覆盖率达标**：测试覆盖率 ≥ 配置中的阈值
- [ ] **QA log 干净**：qa-log.md 中无未回答问题
- [ ] **可追溯**：每个 P0 需求对应实现的代码和通过的测试

### 最终返工循环

如果关卡 #3 失败：
1. 汇总所有问题到一个批次（不要一个一个修）
2. 每个问题分配给正确的 Agent
3. 所有 Agent 修完后重新跑 CI（关卡 #2）
4. 重新跑受影响的子阶段
5. 重新评估关卡 #3

## 关卡总览

| 关卡 | 时机 | 检查内容 | 执行者 | 裁决者 |
|------|------|---------|--------|--------|
| #1 | 阶段 1 后 | 需求质量 | Gatekeeper | Lead |
| #1.5 | 阶段 1.5 后 | 方案一致性 | Gatekeeper | Lead |
| #2 | 阶段 2 后 | 编译 + lint | CI Runner + Gatekeeper | Lead |
| #3 | 阶段 3 后 | 审查 + 测试 | Gatekeeper | Lead |

每个 Gate 的执行流程：Lead spawn Gatekeeper → Gatekeeper 出 Gate Report → Lead 审阅并裁决（通过/返工）。

Gatekeeper 解决的是**质量问题**（检查是否认真），Hook（`pre-spawn-gate-check.sh`）解决的是**合规问题**（Gate 是否被跳过）。两者互补，独立工作。
